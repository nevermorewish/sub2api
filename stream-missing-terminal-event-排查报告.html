sub2api 流式报错排查报告

stream usage incomplete: missing terminal event claude-opus-4-8 /v1/messages · stream
实例 43.130.1.77 · 容器 sub2api · 排查时间 2026-07-07 · 数据窗口 近 48h
🎯 结论：不是 Huanxing 的「OpenAI Compatible → Claude Messages」转换引起的。

真正原因是 上游一道约 30 秒的服务端超时：当 claude-opus-4-8 处理 超大上下文（100KB–600KB）且 prompt 缓存未命中时，"首个 token 之前"的 prefill 耗时超过 30 秒， 上游还没发出 message_stop 就把 SSE 连接干净关闭，sub2api 于是记为 missing terminal event。随后 Huanxing 自动重试、命中缓存、首 token 变快，请求就成功了 —— 所以当天最终只有 2 次 502，这批基本都被重试兜住了。

一、错误是什么

报错位置 backend/internal/service/gateway_service.go:5623，经 handler/gateway_handler.go:873 记录为 gateway.forward_failed。

sub2api 转发流式请求时，边把上游 SSE 逐行写给客户端，边扫描"终止事件"：

终止事件 = event: message_stop   或   data: [DONE]   或   data 里 type=="message_stop"

当上游 SSE 流被 EOF 关闭（scanner.Scan() 返回 false 且无 error），却始终没收到 message_stop 时， 就返回该错误。它发生在 HTTP 200、响应头已发出、流已开始之后，所以不是 5xx 状态码 —— 这解释了"为什么 502 只有 2 次却有 15 条错误"。

② 是"干净断流"，不是 sub2api 自己超时

近 48h 内其它异常字符串统计全部为 0，排除了 sub2api 侧的超时与网络读错：

stream data interval timeout = 0    ← sub2api 自身流间隔超时【未触发】
stream read error            = 0    ← 上游连接读取【未报错】
after disconnect             = 0    ← 客户端断开【未发生】
after timeout                = 0
context deadline             = 0
stream read error = 0 + EOF 关闭 = 上游主动 FIN 干净关闭了连接， 而非 sub2api 超时、也非网络异常。断点固定在 ~30s，与请求体大小无关（100KB 和 629KB 都在 ~31s 断）， 说明是上游一道固定的 30 秒服务端超时。

③ 请求体巨大 + 重试即成功 → 指向"首 token 时间 vs 30s"

  • 失败请求 body 都在 103KB–629KB，超大上下文 → 首 token 前的 prefill 很慢。
  • 每次失败后 1–2 秒内自动重试同一账号，重试几乎都成功 —— 因为命中 prompt 缓存， prefill 变快、首 token 在 30s 内到达（哪怕总耗时 65s 也能跑完）。

  1. Huanxing 把请求（已是原生 anthropic_messages 格式）发给 sub2api，body 173KB。
  2. sub2api 选中账号 20，向上游发起流式请求，返回 200、开始拉流。
  3. 超大 prompt 缓存未命中，上游 prefill > 30s，首 token 迟迟不来。
  4. ~31s 时上游干净关闭连接，未发 message_stop → sub2api 报 missing terminal event，写回兜底错误。
  5. Huanxing 1.4s 后自动重试，仍走账号 20（sticky）。
  6. 重试命中 prompt 缓存，首 token 快速到达，流正常跑完（21.3s）→ 成功。

五、为什么和 Huanxing 的格式转换无关

  • 日志里协议已是 protocol: anthropic_messages，请求进 sub2api 时就是原生 Claude 格式，转换正确。
  • 本错误发生在上游返回的响应流上（少了 message_stop），与 Huanxing 做的请求格式转换不在同一条链路。
  • Huanxing 只是客户端，而且它自己重试恢复了 —— 是"受害者"，不是"元凶"。

六、建议（按性价比排序）

  1. 病根｜放宽上游 30s 超时：让供应商放宽这道服务端超时，或改用 prefill 更快的上游线路。 这几个 9527codeclaude* 是同批采购的 key，超时策略一致。
  2. 绕过｜控制 prompt 体积：Huanxing 侧对超大上下文做裁剪/压缩，把首 token 时间压到 30s 以内即可规避。
  3. 兜底｜sub2api 自动重试：对 missing terminal event 增加一次自动 failover/重试， 降低透传到客户端的报错（当前已靠 Huanxing 重试兜住，可再加一层）。
  4. 定位｜区分 30s 来源：高峰期对这几个账号的上游端点单独 curl -N 大 prompt 复现， 确认是"上游 30s"还是"中间链路 30s"。
