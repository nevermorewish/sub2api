sub2api Claude 请求头生成逻辑

生成日期：2026-05-30；基于当前工作区代码，包含上游 Wei-Shaw/sub2api 合并后的 ddf91e9 相关请求头调整。

结论

当前 Claude 上游请求头不是简单透传客户端 header。核心逻辑是先算出最终 anthropic-beta，按这个 beta 清理 body，再做 billing CCH 签名，最后才创建上游请求并写入 Claude Code mimic 头。

OAuth + mimic Claude Code 的 /v1/messages 路径会跳过客户端 header 透传，并强制使用当前仓库内置的 Claude CLI 指纹头；API-key 和非 mimic OAuth 路径仍保留白名单透传语义。

这些规则来自上游 E:\Claude\sub2apigit 的一组历史修复：目标不是“看起来像普通 Anthropic SDK”，而是同时满足 Claude Code 识别、billing attribution、prompt cache、beta/body 能力对称和 CCH hash 一致性。

messages 路径生成顺序

入口：buildUpstreamRequest，关键顺序在 backend/internal/service/gateway_service.go。

#阶段作用
1OAuth 账号获取或创建 fingerprint后续可覆盖 User-Agent 与 x-stainless-* 指纹头。
2重写 metadata.user_id除 metadata passthrough 开启外，会按账号和 fingerprint 规则重写用户标识。
3syncBillingHeaderVersion将 body 内 billing block 的 cc_version=X.Y.Z 同步到实际 User-Agent 版本。
4computeFinalAnthropicBeta在创建 http.Request 前算出最终 beta；这是后续 body 清理和 CCH 签名的依据。
5sanitizeAnthropicBodyForBetaTokens如果最终 beta 不含 context-management-2025-06-27，删除 body 的 context_management。
6signBillingHeaderCCH所有 body 修改完成后，把 billing block 中 cch=00000 替换成 xxHash64 签名。
7创建上游 request 并设置认证头OAuth 写 authorization: Bearer ...；API-key 写 x-api-key。
8白名单透传或跳过透传OAuth mimic messages 路径跳过客户端 header 透传；其他路径按白名单透传。
9应用 fingerprint / 默认头 / mimic 头补齐 content-type、anthropic-version，OAuth mimic 强制写 Claude Code 指纹头。
10删除旧 anthropic-beta 并写入最终 beta避免客户端原始 beta 或大小写变体残留。
11同步 X-Claude-Code-Session-Id如果请求头已有该字段，用处理后的 metadata.user_id 里的 session id 覆盖。
这个顺序来自 ddf91e9 的重排：先算最终 beta，才能知道 body 里哪些 beta-gated 字段能保留；body 变完之后才能签 CCH。否则 header、body、billing hash 三者会互相不一致。

anthropic-beta 决策矩阵

路径最终 beta 来源是否透传客户端 beta备注
OAuth + mimic + messages + 非 Haiku FullClaudeCodeMimicryBetas() 后再应用 dropSet 否 严格模拟 Claude Code；跳过白名单透传，避免客户端 header 污染。
OAuth + mimic + messages + Haiku oauth-2025-04-20 + interleaved-thinking-2025-05-14 后再应用 dropSet 否 代码中 Haiku 走轻量 beta 组合。
OAuth + 非 mimic + messages getBetaHeader(modelID, clientBeta) 后再应用 dropSet 是 用于真实 Claude Code 客户端或 passthrough 场景。
API-key + 客户端传 beta 客户端 anthropic-beta 后再应用 dropSet 是 保持 API-key 客户端控制权。
API-key + 未传 beta 仅当 InjectBetaForAPIKey 开启且 body 需要 beta 功能时注入 无 否则不主动设置 anthropic-beta。
OAuth + mimic + count_tokens FullClaudeCodeMimicryBetas() + token-counting-2024-11-01，再合并客户端 beta 和 dropSet 是 与 messages mimic 不同，count_tokens mimic 保留客户端 beta 合并语义。
OAuth + 非 mimic + count_tokens 无客户端 beta 时用 CountTokensBetaHeader；有客户端 beta 时补齐 token-counting 是 保证 count_tokens 请求携带 token-counting beta。

Billing block 与 CCH

Billing attribution block 来源：backend/internal/service/gateway_billing_block.go。

{"type":"text","text":"x-anthropic-billing-header: cc_version=2.1.156.{fp}; cc_entrypoint=cli; cch=00000;"}
字段当前算法
{fp}取第一条 user 消息的首个文本块，读取字符索引 4、7、20，不足用 0；计算 SHA256("59cf53e54c78" + chars + version)，取 hex 前 3 位。
cch在最终 body 上计算 xxHash64Seeded(body, 0x6E52736AC806831E) & 0xFFFFF，格式化为 5 位十六进制，再替换 cch=00000。
顺序约束必须先 sanitize body，再签 CCH。 如果先签名再删除 context_management，最终 body 和 hash 不一致，容易触发 third-party 判定。

历史原因：5862e2d8 把 billing block 加进 system[0]，因为真实 Claude Code 总是发送它；e51c9e50 再把 cc_version 与实际 User-Agent 同步，并实现 opt-in CCH 签名。当前代码中 CCH 是否签名仍受设置控制，但一旦开启，必须在所有 body 修改之后运行。

与抓包 JSON 的差异点

对比 captures/claude-code-headers/cli_baseline_2026-05-30.redacted.json 和实时抓包时，以下字段天然会变化，不应按固定字符串比较：

  • Content-Length：由最终 body 字节数决定。
  • x-client-request-id：每次请求生成新的 UUID。
  • X-Claude-Code-Session-Id：由处理后的 metadata.user_id 同步。
  • cch：由最终 body 计算，消息内容、metadata、context_management 是否保留都会改变它。
  • cc_version=2.1.156.{fp} 的后三位 {fp}：由第一条 user 文本内容计算；同一 CLI 版本下也会随消息变化。

需要重点保持稳定的是 CLI 指纹头：User-Agent、X-Stainless-*、X-App、Accept、默认 anthropic-beta 顺序，以及 CCH 签名前后的 body 顺序。

和 E:\Claude\sub2apigit 的旧上游相比，当前项目已经把 CLI 指纹从 2.1.92 / Linux / arm64 / SDK 0.70.0 / Node v24.13.0 更新为 2.1.156 / Windows / x64 / SDK 0.94.0 / Node v24.3.0，并把默认 beta 改为 2026-05-30 抓包顺序。

为什么不是全量透传

  • bdbd2916 说明：客户端透传会把第三方工具自己的 header 混进 mimic 请求，导致同一个请求里同时存在“像 Claude Code”和“不像 Claude Code”的指纹。
  • 6dc89765 说明：很多客户端只做了部分伪装，例如带了 Claude Code UA，但缺 billing block、完整 beta、工具名混淆和缓存断点；本地误判为真 CC 会让上游仍按 third-party 处理。
  • 496469ac 说明：真 Claude Code 客户端又不能粗暴全量改 body，否则会破坏它自己的长 system prompt 和 prompt cache 命中。
  • ddf91e9 说明：body 字段是否保留不能只看模型名，必须看最终发送的 beta 能力集合。
